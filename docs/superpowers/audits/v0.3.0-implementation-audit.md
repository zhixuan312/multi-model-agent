## Task 5: openai-runner continuation fix + reason precision

**Commit:** 2f5063a (feat: task 8 + task 5 continuation budget)

### Finding T5-1: `runContinuationTurn` helper was NOT implemented

**Severity:** High
**Location:** `packages/core/src/runners/openai-runner.ts`
**Detail:** The plan (Step 5-6) defines a `runContinuationTurn` discriminated-union helper that wraps all three continuation call sites and catches `MaxTurnsExceededError` with context, returning `{ ok: true, result }` or `{ ok: false, cause, label, turnAtFailure }`. This helper is **absent** from the implementation.

The three continuation call sites are handled differently:
- **Plan Step 5:** A single `runContinuationTurn` helper called by all three sites
- **Actual implementation:** Each continuation site directly calls `runTurnAndBuffer` with `SUPERVISION_CONTINUATION_BUDGET` as the budget. Example at line 421:
  ```typescript
  currentResult = await runTurnAndBuffer(continueWith(currentResult, warning), SUPERVISION_CONTINUATION_BUDGET);
  ```
  There is NO `MaxTurnsExceededError` catch around these calls. If the model exhausts the 5-turn budget on a continuation, the error propagates to the outer `catch (err)` handler at line 498, which checks `if (err instanceof MaxTurnsExceededError)` and returns `status: 'max_turns'` â€” the **exact misclassification** this fix was supposed to prevent.

The key consequence: a supervision re-prompt that exhausts the 5-turn continuation budget will still be misclassified as `'max_turns'` instead of `'incomplete'`. The `SUPERVISION_CONTINUATION_BUDGET = 5` constant exists but provides **no semantic protection** without the catch.

**Required fix:** Implement `runContinuationTurn` as a module-level helper (or closure) that:
1. Calls `runTurnAndBuffer` with `SUPERVISION_CONTINUATION_BUDGET`
2. Catches `MaxTurnsExceededError` 
3. Returns discriminated union `{ ok: true, result }` | `{ ok: false, cause, label, turnAtFailure }`
4. All three continuation call sites use this helper instead of bare `runTurnAndBuffer(...SUPERVISION_CONTINUATION_BUDGET)`

### Finding T5-2: `buildSupervisionExhaustedResult` does not accept a `reason` parameter

**Severity:** Medium
**Location:** `packages/core/src/runners/openai-runner.ts:640-685`
**Detail:** The function signature is:
```typescript
function buildSupervisionExhaustedResult(
  currentResult: AgentRunOutput,
  scratchpad: TextScratchpad,
  tracker: FileTracker,
  providerConfig: ProviderConfig,
  durationMs: number,
  parentModel?: string,
  traceBuffer?: ProgressEvent[],
): RunResult
```
It does NOT accept an optional `{ reason?: string }` parameter. The `result.error` field is **never populated** on supervision-exhausted paths (lines 656-684 construct the return object without an `error` spread).

Plan Step 7 requires adding the optional reason parameter so call sites can pass precise diagnostic strings like `"supervision reground continuation exhausted the 5-turn sub-budget at turn N"`.

**Required fix:** Add `{ reason?: string } = {}` parameter, spread `...(opts.reason && { error: opts.reason })` in the return object. Update the supervision-exhausted call site at lines 487-495 to pass a reason.

### Finding T5-3: Outer `MaxTurnsExceededError` catch does not populate `result.error`

**Severity:** Medium
**Location:** `packages/core/src/runners/openai-runner.ts:499-531`
**Detail:** The outer `MaxTurnsExceededError` catch at line 499 constructs the return object without an `error` field. The plan (Step 8) requires:
```typescript
error: `agent exhausted user-declared maxTurns limit (${maxTurns}) after ${turnsAtFailure} turns`
```
**Required fix:** Add `error` field with precise reason string to the return object.

### Finding T5-4: Supervision-exhausted call site (lines 487-495) has no `reason` parameter

**Severity:** Medium
**Location:** `packages/core/src/runners/openai-runner.ts:487-495`
**Detail:** The call:
```typescript
const exhausted = buildSupervisionExhaustedResult(
  currentResult,
  scratchpad,

```
Should pass a reason per plan Step 9: `"supervision loop exhausted after ${supervisionRetries} re-prompts (last kind: ${validation.kind ?? 'unknown'})"`.

**Required fix:** Update call site to pass reason. Also update `buildSupervisionExhaustedResult` signature first (see T5-2).
