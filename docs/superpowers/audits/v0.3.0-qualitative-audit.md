### `trimProgressTrace` — ⚠️ QUALITATIVE DEVIATION

**Implementation:** `packages/core/src/runners/supervision.ts:100-246`

**Constants match plan:**
- `TRACE_MAX_EVENTS = 80` ✅ (line 100)
- `TRACE_MAX_CHARS = 16_384` ✅ (line 101)
- `TRACE_DROP_PRIORITY`: text_emission=1, tool_call=2, boundary=100 ✅ (lines 103-111)

**Deviation 1: Algorithm differs from plan**

The plan specifies a priority-dropping algorithm:
1. Sort droppable entries by priority ascending, then index
2. Drop in that order until both limits (event count + char count) are satisfied
3. Fallback: first 10 + last 30 if priority drop fails

The implementation uses a different multi-phase approach:
1. Drop to ≤ 40 events (count-based), tracking dropped kinds
2. String-compact previews/toolSummaries iteratively (halving stringLimit each iteration)
3. Only then check character budget — may not fire if dropping alone brought count under limit
4. If still over 40 events after compaction → first 10 + last 30 fallback

**Consequence:** The implementation may stop dropping text_emission events at 40 (before string compaction reduces their individual sizes). A 40-event trace of 300-char previews = ~12KB + boundary events. This is under 16KB, but barely — a higher character budget event stream would pass without the character limit being enforced as specified.

**Deviation 2: `capExceededByBoundaryEvents` fires when boundary events alone exceed cap, but the plan's intent was different**

The plan's size-bounded test (Step 17) implies: after dropping to ≤ 80 events, if `TRACE_MAX_CHARS` is still exceeded → droppable phase trims character content until under 16KB. The implementation's string compaction phase serves this role but may not trigger if the event-drop phase already succeeded by chance.

**Severity:** Low. The never-drop boundary events policy is upheld. The capExceededByBoundaryEvents mechanism correctly signals when boundary events consume the budget. The actual behavior is conservative (fewer drops than maximum, more string compaction).

## Task 2 (injection types): `injectionTypeFor` — ⚠️ QUALITATIVE GAP

**Implementation:** `packages/core/src/runners/injection-type.ts`

### What was implemented

A shared `injectionTypeFor` helper maps `DegenerateKind` → injection type label:

```typescript
// injection-type.ts
export function injectionTypeFor(kind: DegenerateKind | undefined):
  'supervise_empty' | 'supervise_thinking' | 'supervise_fragment' {
  switch (kind) {
    case 'empty':        return 'supervise_empty';
    case 'thinking_only': return 'supervise_thinking';
    case 'fragment':
    case 'no_terminator': return 'supervise_fragment'; // combined
    default:              return 'supervise_fragment'; // catches undefined + 'insufficient_coverage'
  }
}
```

### Deviation: `insufficient_coverage` maps to `supervise_fragment`

**The problem:** When `validateCoverage` fails, the runner sets `validation.kind = 'insufficient_coverage'` (lines 467-471 of openai-runner.ts). Then `injectionTypeFor('insufficient_coverage')` returns `'supervise_fragment'` (the default). The emitted progress event says `injectionType: 'supervise_fragment'` but the re-prompt text is coverage-specific.

**Plan specification:** The plan's `ProgressEvent` type (types.ts) lists 7 injection types including `supervise_insufficient_coverage`. The plan's `buildRePrompt` switch (supervision.ts) has a dedicated `case 'insufficient_coverage'`. The intended behavior is a distinct `injectionType: 'supervise_insufficient_coverage'` event.

**What the plan said:**
- ProgressEvent injection types include: `supervise_empty | supervise_thinking | supervise_fragment | supervise_no_terminator | supervise_insufficient_coverage` (from the plan's ProgressEvent type listing)
- The `injectionTypeFor` function in the plan's design collapses `fragment` and `no_terminator` into `supervise_fragment` — this is the documented and implemented behavior ✅
- But `insufficient_coverage` should have its own type

**The gap:**
1. `ProgressEvent.injectionType` union does NOT include `'supervise_insufficient_coverage'` — only the 3 types above ✅ (matches plan's actual implementation)
2. `injectionTypeFor` falls through `insufficient_coverage` → `supervise_fragment` ❌ (should return a distinct type, or the ProgressEvent type should be extended)

**Severity:** Medium. The re-prompt message is correct (coverage-specific). Only the `injectionType` label on the progress event is misclassified. Callers observing the progress trace would see `supervise_fragment` for coverage failures, not a coverage-specific type.

**Note:** The design justification (code comment in `injection-type.ts`) explains the `fragment`+`no_terminator` collapse as intentional. A similar justification could apply to `insufficient_coverage`, but it wasn't made — the `insufficient_coverage` defaulting to `supervise_fragment` appears unintentional.
