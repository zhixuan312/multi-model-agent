## [unreleased] - 2026-05-02

### Fixed
- **Self-heal stuck telemetry queues with rotated installIds.** The flusher now drops contiguous head-prefix records whose `installId` does not match the current local identity, in addition to the existing legacy-schema drop. The single per-flush identity snapshot is threaded into upload as a parameter; mismatched records are dropped locally without a network round-trip. After `mma telemetry reset-id`, any queued events from the previous identity are dropped on next flush. (Bug 1)
- **Exponential backoff actually doubles.** Removed the `clearBackoff()` call inside `#scheduleBackoff` and the `#backoffMs = 0` reset inside the timer callback. Stuck installs now retry 5/10/20/40/60 (cap) min instead of 5 min flat — drops backend load from 12 req/hr to ~1 req/hr at steady-state. (Bug 2)
- **`claude-agent-sdk` 0.2.113+ env replace.** The claude-runner's per-invocation `env` block now spreads `...process.env` first so `PATH`, `HOME`, proxy vars, and locale vars survive into the spawned Claude subprocess. The next routine `npm install` will resolve `^0.2.112` to `0.2.126`; without this fix, `claude-compatible` providers (DeepSeek, OpenRouter, etc.) would silently lose subprocess env vars on upgrade. (Bug 3)
- **OpenAI reviewer findings preserve severity.** Replaced the OpenAI provider's JSON-block extraction with `Agent.outputType` (Structured Outputs). `parsedFindings: AnnotatedFinding[] | null` is now a required field on `RunResult`. Reviewer-emitted critical/high/medium/low severities round-trip without collapsing to medium via `parseReviewerFindings` fallback. Claude/Codex paths unchanged. (Bug 4)
- **Severity vocabulary locked to 4-tier.** Widened `DiffReviewConcern.severity` to `'critical' | 'high' | 'medium' | 'low'`; corrected a stale `'major' | 'minor'` doc comment on `RawConcern.severity`; added a contract test that fails CI if a future PR adds, removes, or renames a tier. (Bug 5)
- **Fallback-unavailable preserves implementer work.** When both standard and complex tiers transport-fail, the lifecycle now finalizes the implementer stage from the higher-work-score salvage source so the emitted `task.completed` event carries the real `duration_ms`, `turns`, `files_written`, `cost`, and `implementer_model` instead of `duration_ms=2 / turns=1 / tokens=0 / cost=0 / stages=[] / implementer_model='custom'`. R2.1 (empty stages → only allowed for `brief_too_vague|error`) is unchanged — the fix populates stages, not the allow-list. (Bug 6)
- **Server survives parent pipe close.** `serve.ts` startup now installs `process.stdout.on('error')`, `process.stderr.on('error')`, `process.on('uncaughtException')`, and `process.on('unhandledRejection')` listeners. EPIPE-coded errors `exit(0)` (cleanly — no consumer left); other errors `exit(1)` after best-effort diagnostic emission. Wires the existing `ShutdownCause = 'stdout_epipe' | 'stdout_other_error' | 'uncaughtException' | 'unhandledRejection'` enum values to actual code paths. Listeners are removed in `stop()` so in-process test boots don't accumulate them. (Bug 7)
